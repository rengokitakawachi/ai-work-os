# タイトル案

handover は読むだけでは足りない  
再開品質を上げるには「正本への接続」まで定義する

## 問題

引き継ぎ書を用意しても、次のスレッドで handover を読んだだけでは十分に再開できないことがある。  
handover は要約であり、仕様そのものでも実行順そのものでもないため、その後にどの正本へ戻るかが曖昧だと、再開のたびに確認漏れや判断ぶれが起きる。

## 背景

スレッド切替後の再開フローを整理する中で、最新 handover の特定と読取だけでは足りず、関連 docs / notes / code の読取、さらに operations の確認まで行わないと「今どこから再開すべきか」が安定しないことが分かった。  
特に、handover には Issues / Next Actions がある一方で、短期実行順の正本は operations であるため、両者の接続ルールが必要だった。

## なぜ重要か

- スレッドをまたいでも再開品質を安定させられる
- 要約と正本を混同せずに済む
- 仕様判断と実行順の参照先が明確になる
- AI に再開させるときの手順を定型化できる

## 解決

再開開始ルールを「handover を読む」で終わらせず、以下まで含めて定義する。

1. notes tree から最新 handover を特定する
2. handover を read する
3. Related docs / Related notes / Related code を読む
4. operations を読み、短期実行順の正本を確認する
5. Issues / Next Actions をもとに、次の着手 1 件を確定する

## 学び

handover は入口としては強いが、それ自体は正本ではない。  
本当に必要なのは、handover を読み終えたあとに

- 仕様は docs
- 実行順は operations

へ戻ることだった。  
再開品質は、要約の質だけでなく、正本への接続設計で決まる。

## 意思決定

- 再開時は handover 探索から始める
- handover を読まずに状況把握を始めない
- handover は入口、仕様判断は docs、実行順は operations を正とする

## Before / After

- Before:
  handover を読めば再開できるはずだと考えていた

- After:
  handover の後にどの正本へ戻るかまで定義しないと再開品質は安定しないと分かった

## メモ（ラフ）

- handover は「圧縮された入口」であって「判断の終点」ではない
- 要約と正本を分ける設計
- AI に再開させるなら、入口 → 接続先 → 次着手確定まで必要
- operations を見ずに作業しない原則との接続が重要
