# タイトル案

ADAM の開発は開発支援ではなく OS 骨格開発に近い  
個別機能ではなく共通オペレーティングモデルを先に作る

## 問題

一見すると ADAM の開発は開発支援機能の追加に見えるが、その見方だけだと、今やっている整理の本当の価値が見えにくい。  
実際には、個別機能よりも先に、AI Work OS 全体に効く骨格を整備している側面が強い。

## 背景

ここまでの整理では、

- 正本の定義
- handover を使った再開
- intake routing
- issue routing
- roadmap / plan / operations
- review
- 正本と view の分離

のような、システム全体に効く流れや役割分離を先に定義してきた。  
これは ADAM 専用の便利機能ではなく、将来的に EVE にも展開できる共通オペレーティングモデルの原型と見た方が自然だった。

## なぜ重要か

- 個別改善と骨格整備を区別して優先順位を決められる
- EVE へ展開できる共通部分を見つけやすい
- ADAM 開発の意味づけが明確になる
- 今の作業を Phase 0 として位置づけやすくなる

## 解決

ADAM の開発を、単なる開発支援ではなく AI Work OS の骨格整備として捉える。

- ADAM 固有の便宜機能か
- OS 全体の骨格か
- EVE にも展開できるか

を判断基準に置くことで、今やるべきことを見失いにくくする。  
この整理をもとに、`Phase 0: Common Operating Model Foundation` を plan として置く。

## 学び

ADAM の開発フローは、開発専用ローカルルールではなく、AI Work OS 全体に展開できる骨格の原型として見た方が一貫性が高い。  
感覚としては、アプリ機能を足しているというより、OS のカーネルを先に作っている状態に近い。

## 意思決定

- 今の ADAM 開発は AI Work OS の骨格開発として扱う
- 個別機能追加より、共通 operating model の整備を優先する
- その意味づけを Phase 0 plan として保持する
- 今後は「ADAM専用か、OS骨格か、EVE展開可能か」で判断する

## Before / After

- Before:
  ADAM の開発を開発支援機能の積み上げとして見がちだった

- After:
  ADAM の開発は AI Work OS 全体の骨格整備であり、Phase 0 として扱うのが自然だと整理できた

## メモ（ラフ）

- 「OSのカーネルを作っている」に近い
- 共通化できる部分を先に作る
- EVE 実装前の foundation として意味がある
- roadmap / plan / operations に Phase 0 を置く理由になる
