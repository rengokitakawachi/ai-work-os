# タイトル案

フォルダ移行は見た目ではなく SHA で確認する  
README差分と実データ差分を分けて扱うリポジトリ整理の実務

## 問題

フォルダ移行やリネーム作業では、見た目上は同じに見えても、実ファイルの差分や不可視差分が残っていることがある。  
README の追加や削除、`.gitkeep`、特殊ファイル名、末尾改行などが混ざると、「移行できたつもり」で古いフォルダを消して事故る。

## 背景

notes の indexed 構造への移行では、旧フォルダと新フォルダが大量に併存した。  
このとき、単純にファイル数や見た目で判断すると危険だったため、SHA 一致をベースに移行完了を判定した。  
さらに、README の有無は構造説明の差分であり、実データ差分とは分けて扱う必要があった。

## なぜ重要か

- 大規模フォルダ整理で誤削除を防げる
- 「論理上は移行済み」と「実際に消してよい」を分けて判断できる
- AI に整理を任せるときも、確認基準を機械的にできる
- README 追加のような説明用差分に振り回されず、実データ本体に集中できる

## 解決

移行判定を以下の順で行う。

1. 旧新フォルダの対応関係を固定する  
2. README を比較対象から除外するかどうかを先に決める  
3. 実ファイル単位で SHA を照合する  
4. 一致 / 不一致 / 比較不能 に分類する  
5. 削除可能かどうかを「一致判定」と「削除権限」で分けて判断する

## 学び

- フォルダ移行の本質は名前変更ではなく、実ファイル一致の確認
- README は説明のために足してよいが、移行完了判定とは切り分けるべき
- 「削除してよい」と「AI が今削除できる」は別概念
- 特殊ファイル名や不可視差分は最後まで残件になりやすい

## 意思決定

- フォルダ一致判定は README の有無を除外して見てよい
- 実データ一致ベースで旧フォルダの削除可否を判断する
- delete 制約がある場合は、削除可能性と実行可能性を分けて扱う

## Before / After

- Before: 見た目や件数で「たぶん移行済み」と判断しそうになっていた
- After: SHA ベースで一致確認し、削除判断を機械的に説明できるようになった

## メモ（ラフ）

- notes の indexed 移行で実際に使った手順
- README差分のみで不一致に見えるケース
- `.gitkeep` や特殊ファイル名で詰まるケース
- inbox は最後に4本まで残件を絞れた
- AI運用では「論理」と「ツール制約」を分けるのが重要
