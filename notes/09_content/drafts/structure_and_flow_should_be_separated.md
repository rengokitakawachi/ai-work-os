# 構造とフローは分離すべきか

## 問題

知的生産システムや開発フローを設計するとき、
「構造」と「フロー」を同じレイヤーで扱ってしまうことが多い。

その結果、

- フォルダ構成がそのまま手順書になる
- 手順が変わるたびに構造が揺れる
- 構造を整理したいのか、実行順を整理したいのか分からなくなる

という問題が起きる。

## 背景

開発フローを整理する中で、issue / design / operations / future / archive などのレイヤーを定義していくと、
「どこに置くか」と「どう流れるか」は似ているようで別問題だと分かった。

たとえば、

- issue は問題を持つ場所
- design は解決を考える場所
- operations は実行する場所

であり、これは構造の話である。

一方で、

- issue → design → operations → 実行

というのはフローの話である。

この2つを混ぜると、構造の責務が曖昧になりやすい。

## なぜ重要か

この違いを意識しないと、システムはすぐに崩れる。

構造だけをきれいにしても、実行の流れが接続されていなければ運用できない。
逆に、フローだけを整えても、各レイヤーの責務が曖昧なら情報が混ざって崩壊する。

つまり重要なのは、

- 構造は分離する
- フローは接続する

この両立である。

## 解決

レイヤーごとの責務を明確に分ける。

- issue = 何が問題か
- design = どう解くか
- operations = 何をやるか
- future = 今やらないもの
- archive = 過去の保持

そして別軸として、フローを定義する。

- inbox
- intake review
- classification
- issue / design / knowledge / future
- operations
- 実行

このように整理すると、

- 構造は崩れにくくなる
- フローは理解しやすくなる
- 実行判断も安定する

## 学び

「分離」は重要だが、それだけでは足りない。

本当に必要なのは、
分離された構造同士がどの順番で接続されるかを設計することだった。

構造をきれいにするだけでは運用できない。
フローを作るだけでは再利用できない。

この2つを分けて考えることで、はじめてシステムになる。

## 意思決定

- 構造とフローは別物として扱う
- issue / design / operations はレイヤー定義である
- それらの間の移動はフローとして設計する
- 分離と接続を両立させる

## Before / After

- Before:
  構造と手順が混ざっていた

- After:
  構造は責務で分け、フローは別で定義するようになった

## メモ（ラフ）

- 「構造を見る」とは何か
- なぜ多くの人は構造とフローを混ぜてしまうのか
- Notion / Obsidian / Todoist でも同じ問題が起きる
- 記事の締めで「分離し、接続せよ」と言える
