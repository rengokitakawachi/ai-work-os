# Handover

## 概要
本セッションでは、ADAM の運用構造整理と、EVE 価値説明の素材整理を進めた。

前半では、conversation-triggered routing を issue 中心モデルへ再設計し、operations は issue から派生する実行単位として扱う方針を確定した。さらに、active-first execution と reroll before execution を運用原則として整理し、active / next / archive の operations 3層を再構築した。

後半では、ADAM instruction が冗長化していた問題に対し、instruction と knowledge の分離を実施した。`adam_instruction.md` は行動拘束ルールに絞って短縮し、背景知識・用語・詳細運用仕様は `adam_knowledge.md` に分離した。

別件として、EVE の価値を一般向けに説明するスライドの要点整理も進めた。主役は EVE とし、ADAM は後段で「EVE を開発するための先行システム / プロトタイプ」として出す流れがよいと整理した。ユースケースは「EVE と会話して仕事が進む世界」で統一し、特に以下の3本が有力と判明した。
- 朝「おはよう」で1日の予定とタスクが整う
- 急な締切変更・突発タスクでも、伝えるだけで今日の動きが再構成される
- 会話の中で出た課題や気づきが、そのまま整理され、必要ならタスク化される

ただし、EVE価値説明はまだ最終スライド構成・完成原稿まで落とし切っていない。

---

## 成功（Success）
- conversation-triggered routing を issue 中心モデルへ再設計した
- conversation routing の粒度ルール、提案フェーズ、source_ref、rolling 接続を notes design に反映した
- active-first execution / reroll before execution を運用原則として確定した
- `active_operations.md` を現行状態に合わせて再構築した
- `next_operations.md` を現行 schema / execution governance に合わせて再構成した
- `archive_operations.md` に旧 active から外れたタスクを移動し、履歴整合を回復した
- `adam_instruction.md` を「行動ルールのみ」に短縮した
- `code/config/ai/adam_knowledge.md` を新規作成し、背景知識を分離した
- EVE を主役にした一般向け価値説明の方向性を整理した
- EVE の具体的ユースケースとして「朝の挨拶」「突発対応」「会話からタスク化」の3本柱が有力と判明した

---

## 判明事項（Findings）
- 会話起点の新論点は原則 issue として受ける方が notes 全体の構造と整合する
- operations は起点ではなく、issue から派生する実行単位として扱う方が崩れにくい
- 会話中の新規タスク候補は、その場で横入り実行せず reroll を通すべき
- active_operations は「今やる順」の正本として扱い、実行は上から順に行うべき
- instruction は「挙動を拘束するルール」に絞り、背景知識は別ファイルへ分離した方が保守性が高い
- EVE 価値説明では ADAM を先に出すより、EVE の価値を先に出した方が理解されやすい
- 非エンジニア向け説明では、抽象説明より「会話 → 何が起きるか」の具体ストーリーが有効
- EVE の世界観は「日常業務は EVE と会話して進める世界」で統一した方が強い
