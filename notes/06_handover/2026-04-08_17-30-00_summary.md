## 概要
このセッションでは、最新 handover と関連 docs / notes / code を再読して現状把握を行い、`active_operations.md` の stale な先頭 task を整合回復した。その後、active-first execution 原則の基本ケースを検証し、会話起点の論点を issue / design / next に流す実運用を一部試した。途中で、元々のテスト目的であった「会話中に発生した新規課題を operations に入れてから実行できるか」の full flow は未達であり、厳密には失敗と整理した。さらに、会話の勢いで operations 原則を忘れやすい欠陥に対して、`adam_instruction.md` を更新し、新規課題・新規作業候補が出たときは実行や保存の前に判定と配置提案を必ず挟む停止ルールを追加した。加えて、EVE に向けた decision 抽出履歴モデルのアイデアを issue 化し、ADAM で先行試験する task を `next_operations.md` に追加した。

## 判明事項（Findings）
- active-first execution は、少なくとも「stale な active を直してから、先頭 task を進める」基本ケースでは成立する
- reroll before execution は、毎ターン無条件に要るわけではなく、新規候補発生時の再配置ルールとして扱うのが自然
- ただし ADAM は会話に流れ込むと operations 原則を忘れやすく、instruction に停止ルールを入れないと実運用で崩れやすい
- 複数論点が並行すると、進め方提案を先に行わない限り、会話の勢いで一論点に流れ込みやすい
- decision は EVE で重要になりうる。特に
  - なぜそう決めたか
  - 何を前提に決めたか
  - 他にどんな案があったか
  の保持が重要
- decision の集約モデルは、docs / issue / design / plan / operations / dev_memo を紐づけ元とし、`notes/05_decisions/` を集約先にする設計仮説が有力
- ただし decision 論点はまだ active に繰り上げておらず、operations 原則に従って next に置いた
