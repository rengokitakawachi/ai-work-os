# Handover

## 概要
本セッションでは、Phase 0 の中核である common operating model の整合を前進させた。

主な到達点は以下。

- routing と review の分離を前提に、Flow Control を上位概念として整理
- intake routing / issue routing / operations rolling を共通骨格上の usecase として再定義
- operations rolling を「候補を優先順位で並べ、7日枠に入るものを active_operations とする」モデルへ確定
- future を「未来に関係する要素を保持するレイヤー」として時間軸で定義
- operations task schema を定義し、rolling_day / due_date / why_now / source_ref の扱いを整理
- active_operations は schema 準拠の最新状態になっていることを確認
- Supabase 論点は本流から外し、dev_memo に軽量退避した

---

## 成功（Success）
- Flow Control and Usecase Architecture を design として保存
- Operations Rolling Generation and Prioritization Spec を ranking + 7日枠モデルへ更新
- future_layer_operating_spec を routing / review 分離に整合させた
- standard_development_flow_v2 を future の時間軸定義と複数流入元の operations 候補生成に合わせて更新
- operations task schema を新規作成
- manual rolling round 1 の観測メモを保存
- active_operations は schema 準拠状態で最新化済み
- instruction に Flow Control / ranking based operations rolling / routing再構成ルールを反映済み
- Supabase は metadata DB 候補として dev_memo に退避済み

---

## 判明事項（Findings）
- operations は issue / design だけでなく、plan / dev_memo / 会話 / review からも候補生成される
- operations の配置は固定条件ではなく、相対順位と7日枠で決める方が自然
- スコアは決定ロジックではなく補助情報として扱うべき
- future は「今やらない箱」ではなく、未来に関係する要素を保持するレイヤー
- future/inbox, future/issue, future/design, future/plan は時間軸で定義すると整合がよい
- operations は日付や due を持ってよいが、schedule 正本にはしない
- Outlook は schedule 正本、Todoist は execution view、operations は短期実行順正本という分離が安定
- handover 作成前時点で active_operations は schema 準拠の内容になっている
- operations_generation_rules.md の再更新は必要だが、repo 上の read 結果では旧内容が残っており、再度整合確認して更新する必要がある
