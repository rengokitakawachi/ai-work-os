# Handover

## 概要
本セッションでは、Phase 0 の common operating model を docs / design / operations レベルでほぼ整合させたうえで、
次の中核テーマとして「ADAM との会話から新しい issue / design / dev_memo / operations / future をどう生成し、rolling 提案までつなげるか」を設計対象として立ち上げた。

主な到達点は以下。

- operations rolling を generation / ranking / placement の一体処理として設計確定
- old `operations_generation_rules.md` を rolling spec に統合し、archive/design へ退避
- `standard_development_flow_v2.md` を current model に整合
- future を「保管箱」ではなく「時間軸レイヤー」として再定義
- `docs/15_notes_system.md` を current notes model に更新
- 会話起点の candidate routing / rolling を新規 design として起票
- ただし、会話起点フローは「単一分類」ではなく「主分類 + 派生出力」が必要と判明し、その前提で設計を開始した

今回の重要判断は、
ADAM との会話から発生する新論点は、1つの保存先に閉じず、
issue / design / operations candidate を並行して生成しうることを正式に扱う必要がある、という点にある。

## 判明事項（Findings）
- operations は issue / design / plan / dev_memo / 会話 / review から候補生成される
- operations candidate は終点ではなく、issue や design を並行して生成しうる
- 会話起点の論点は `1論点 = 1保存先` ではなく `1論点 = 複数レイヤー展開` が自然
- future は「今やらない箱」ではなく、時間軸で配置するレイヤーとして扱うべき
- docs/15 の更新により、notes system の docs 側は current model にかなり追いついた
- スレッド移行タイミングは token 量より「構造イベント」で判断する方がよい
- ADAM は自動移行そのものではなく、自動提案として thread handover を促す設計が自然
- 会話起点フローを instruction に入れる前に、主分類・派生出力・rolling 提案・合意後保存の厳密な定義が必要
