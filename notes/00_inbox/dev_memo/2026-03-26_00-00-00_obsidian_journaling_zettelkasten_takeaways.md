- Zettelkasten ベースでノート種別を分ける

強いのは思考の深掘りと保存前の対話設計。

要するに、"何を保存するべき状態まで思考を育てるか" の整理に強い。

## 共通して重要な思想

### メモは保存して終わりではない

両記事とも共通して、書くだけでは価値が薄いという前提に立っている。

必要なのは以下。

- 見返せること
- 再利用できること
- 次の思考やアウトプットにつながること

### AIの役割は保存より加工

AIは正本ではなく、以下の補助に置くのが安定している。

- 要約
- 分析
- タグ補助
- 関連候補生成
- アウトプット変換
- 壁打ち

## 構造面で参考になること

### ノート種別の責務分離

Zettelkasten 記事で出てくる分類はそのまま採用する必要はないが、責務分離の考え方は有効。

- LiteratureNote = 外部情報の取り込み
- FleetingNote = 断片思考
- PermanentNote = 再利用可能な整理済み知見
- Daily = 日次メモ
- Memo = さらに軽い一時メモ
- IndexNote = 目次 / 索引

本質は、未整理 / 一時 / 整理済み / 索引 を分けること。

### リンク中心の構造

両記事とも、フォルダだけでなく `[[リンク]]` と関連候補でノートをネットワーク化している。

- フォルダ = 保管場所
- リンク = 知識関係
- タグ = 横断検索軸

この分離はかなり重要。

### タグは検索補助軸

タグは構造の主軸ではなく探索の補助軸として使うのが良い。

役割:
- ノート横断検索
- 同種ノートの集合化
- まだリンクしていないノートの再発見

## フロー面で参考になること

### 2フェーズ設計

ジャーナリング記事の最重要ポイント。

- フェーズ1: 壁打ち・深掘り
- フェーズ2: 整形して保存

未整理な思考をそのまま確定情報として保存しない、という境界設計が良い。

### 終端シグナルの明確化

「以上です」で保存へ切り替える設計は参考になる。

思考中と保存対象の境界を明示できるため、途中ノイズまで構造化しなくて済む。

### 保存フォーマットの標準化

ジャーナリング記事の保存フォーマットは再利用しやすい。

- 要約
- キーテーマ
- 深掘りポイント
- 印象的なフレーズ
- 関連ノート候補
- 元の記録

特に、原文・解釈・次の論点が分かれている点が良い。

## 運用面で参考になること

### Input / Manage / Output の分離

Obsidian in Cursor 記事は、入力・管理・出力を混ぜない設計になっている。

- Input: Web Clipper / Kindle / LINE / GitHub
- Manage: Zettelkasten / タグ / GitHub / S3
- Output: PermanentNote / Slidev / Anki / audiobook / Calendar

これはそのままツール採用するというより、責務分離の原則として有効。

### 同期戦略は構造とセットで考える

GitHub同期のメリットだけでなく、コンフリクトやモバイル不安定も考慮している点は参考になる。

つまり、保存構造だけではなく同期制約込みで設計する必要がある。

### モバイル入力は役割を絞る

全環境で同じことをやろうとしない。

- PC: 深掘り・自動保存
- Mobile: 閲覧または軽量入力

この分離は運用安定に効く。

## 2本を組み合わせたときの示唆

もっとも参考になる接続は以下。

1. 断片思考を話す / 書く
2. AIと壁打ちして解像度を上げる
3. 構造化して保存する
4. 後で整理済み知見へ昇格させる

つまり、

断片思考 → 対話で深掘り → 構造化して保存 → 再利用可能ノートへ昇格

という流れが作れる。

## そのまま採用しない方がよい点

### notes を正本にしない

AIが付けた要約・タグ・関連候補は補助情報であって正本ではない。

### ノート種別を増やしすぎない

分類は細かくしすぎると運用負荷が上がる。最初は最小構成がよい。

### Daily に何でも入れない

日次ログ・深掘り思考・週次レビューを全部同じ箱に入れると責務が曖昧になる。

## AI Work OS に引きつけた示唆

### 参考にすべき骨格

- ジャーナリング記事 = 思考を深める前段設計
- Zettelkasten記事 = 深まった思考を再利用可能に保存する後段設計

### 位置づけ

- inbox = 未整理入力
- exploration = 断片思考 / 壁打ち結果
- design = 再利用可能な仮説 / 構造化草案
- docs = 最終仕様

### 重要な境界

- 思考の事実 と AIの解釈 は分ける
- links は関係、tags は検索補助、folders は保管場所
- notes は思考資産、docs は仕様正本
