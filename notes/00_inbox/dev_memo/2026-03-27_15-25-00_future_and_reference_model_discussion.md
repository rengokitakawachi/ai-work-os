# 2026-03-27 Future and Reference Model Discussion

## 概要

標準開発フローの上流設計に関連して、future レイヤーの必要性、phase を基準にした inbox / future の振り分け、ファイル単位ではなく論点チャンク単位での処理、source_ref の考え方、相互参照の必要性について整理した。

本メモは会話由来の未整理入力メモであり、正式仕様ではない。

---

## 1. future フォルダの必要性

### 結論

future フォルダは必要と整理した。

ただし、future は「整理済み保管」ではなく、「未来向け未整理保管」として扱う。

### 背景

inbox を仕分けしたときに、現時点では解釈や課題化を行うべきではないが、将来のために残しておきたい情報が存在する。

現時点で無理に解釈すると、判断ミスや premature な issue 化 / design 化につながる可能性が高い。

### 定義

```text
inbox:
近い将来に整理する未整理入力

future:
かなり先で使う前提の未整理入力

archive:
役目を終えたものの退避先
```

### 位置づけ

future は archive ではない。

future は「まだ役目が来ていない未整理資産」を保管する。

---

## 2. phase を基準にした inbox / future の振り分け

### 結論

inbox と future の振り分け基準は、時間感覚ではなく phase を使う方針が自然と整理した。

### 採用ルール

```text
- 現フェーズに関係するもの → いま扱う
- 次期フェーズに関係するもの → いま一次判定する
- それより先のもの → future へ送る
```

### 定義の更新

```text
inbox:
現フェーズまたは次期フェーズで使う未整理入力

future:
次期フェーズより先で使う未整理入力
```

### 効果

- 近い将来 / 遠い将来の曖昧さを減らせる
- roadmap と接続できる
- phase を現在地だけでなく運用判断にも使える
- future に送る基準を明文化できる

---

## 3. 1ファイル内に複数フェーズの情報が混在する問題

### 懸念

web や memo の 1 ファイルの中に、複数フェーズに関する情報が混ざる場合がある。

そのため、ファイル単位で inbox / future を移動するだけでは不十分。

### 結論

処理単位はファイルではなく、論点チャンクとする必要がある。

必要なら 1 ファイルの中身を複数テーマに分割して、それぞれ別の行き先へ送る。

### 方針

```text
- 入力はフォルダ単位で読む
- 処理単位は論点チャンク
- 出力単位は 1テーマ1メモ
- phase 判定はテーマ単位で行う
- future に送るのもテーマメモ単位
- 元ファイルは archive / delete を判断する
```

### 処理イメージ

```text
元ファイル 1
↓
テーマA
テーマB
テーマC
```

各テーマについて、現フェーズ / 次期フェーズ / それ以降 を判定する。

---

## 4. インテークレビューの更新定義

### 結論

インテークレビューは、未整理入力フォルダ内のファイル群を全体読取し、内容を論点チャンクに分解し、テーマ単位に統合したうえで、各テーマについて phase 適合、一次判定、保存先を判断する工程として再定義するのが自然と整理した。

### 更新版定義

```text
インテークレビュー:
未整理入力フォルダ内のファイル群を全体読取し、
内容を論点チャンクに分解し、
テーマ単位に統合したうえで、
各テーマについて
- phase 適合
- 一次判定
- 保存先
を判断する工程
```

### 出力単位

インテークレビューの出力は 1テーマ1メモ とする。

## 5. source_ref の考え方

### 論点

入力ファイルを論点チャンクに分割し、テーマメモを作る場合、元の文脈や出自をどう保持するかが課題となる。

### 結論

source_ref はかなり有用であり、Obsidian 的な「元ノートとのつながりを残す」思想に近いと整理した。

### 定義

```text
source_ref:
このテーマメモが、
どの元ファイル・元入力から切り出されたかを示す参照情報
```

### 位置づけ

source_ref は通常の関連リンクというより、出典や出自を示す参照である。

### 効果

- 元の文脈に戻れる
- 1つの入力から複数テーマへ分割しても追跡できる
- future へ送った論点の再解釈がしやすい
- issue / design の根拠を辿れる

### 補足

複数の元ファイルを統合してテーマメモを作る場合、source_ref は複数持ってよい。

---

## 6. 相互参照の必要性

### 結論

相互参照はゼロにはしない方がよい。

ただし、最初から全面採用する必要はない。

### 優先順位

```text
最優先:
source_ref

次点:
明確な親子関係の参照
- issue → design
- plan → operations

後回し:
関連論点どうしの自由リンク
```

### 方針

- まず source_ref を採用する
- 次に明確な接続だけリンクする
- それ以外の相互参照は必要になってから増やす

### 判断

相互参照は必要だが、全面リンクより「出自追跡」と「明確な接続」の方を優先する。

---

## 7. 現時点の時間軸レイヤー整理

```text
inbox:
現フェーズ / 次期フェーズ向け未整理入力

future:
それ以降のフェーズ向け未整理入力

current core:
issues / design / operations / plans

archive:
役目を終えたもの
```

### future から戻す条件

future review により、その内容が現フェーズまたは次期フェーズに入ったと判断できたとき、inbox または中核レイヤーへ戻す。
