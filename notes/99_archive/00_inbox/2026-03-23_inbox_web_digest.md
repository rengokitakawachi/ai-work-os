## 横断的に見えた共通原則

### 1. 小さく始める

いきなり多機能化しない。
まず単一ユースケースで有効性を確認する。

### 2. 中間成果物を明示する

いきなり完成品に行かず、
- ナレッジ整理
- 行動ルール
- 権限設計
- DESIGN.md のような設計ルール
のような中間レイヤーを持つと品質が安定する。

### 3. ログと改善を最初から組み込む

AI機能は導入後の観測と改善が前提。
ログ、フィードバック、更新責任の設計が必須。

### 4. ナレッジ品質が出力品質を決める

古い情報、重複、誤情報を整理しないとエージェント品質は上がらない。

### 5. 入口は自由、昇格は厳格

inbox は粗くてよい。
ただし exploration / design / docs に進むほど構造化と厳密さを上げる。

## AI Work OS への示唆

### A. notes/inbox/web の整理方針

このディレクトリは「外部情報の原文置き場」として有効。
ただし放置するとノイズ化するため、定期的に以下へ昇格させる運用が必要。

- 汎用知見 → notes/exploration
- 仕様草案化が必要なもの → notes/design
- 実装タスク化が必要なもの → notes/backlog/dev-backlog.md

### B. AIエージェント設計原則の再確認

特に次は現行思想と整合する。

- 単一ユースケースから始める
- docs / knowledge を整備してから精度を求める
- 人間承認を前提にする
- ログを残す
- 改善責任者または改善フローを持つ

### C. DESIGN.md 的レイヤーの導入検討

Stitch 記事の本質は UI ツール紹介よりも、
「曖昧な意図を再利用可能な markdown ルールへ落とす」点にある。

AI Work OS でも以下のようなテンプレート化を検討できる。

- design/agent_behavior.md
- design/ui_rule.md
- design/interaction_rule.md
- design/output_contract.md

つまり、実装前に人とAIの共通解釈を固定する markdown 成果物を置く設計である。

### D. Obsidian 的 inbox 運用の妥当性

現行の notes/inbox は妥当。
ただし inbox の価値は「溜めること」ではなく、
後段の exploration / design へ変換していくことにある。

## 今回の結論

今回の inbox/web から最も有益だった知見は次の3点。

1. AIエージェント導入は「小さく始める」「人間承認」「ログ改善」が核
2. DESIGN.md のような markdown ベースの中間成果物は、AI と実装の解釈ズレを減らす
3. inbox は原文の保管庫として有効だが、価値は exploration / design への昇格運用で決まる
