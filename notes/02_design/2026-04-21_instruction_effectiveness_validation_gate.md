# 2026-04-21 instruction_effectiveness_validation_gate

## 目的

今回追加した一般化した再発防止ゲートが、
実際の判断で効果を持つかを検証する枠組みを固定する。

ここで検証したいのは、
`instruction に追記できたか`
ではなく、

- task 提案
- completed 判定
- reroll / daily review

の実判断で、
混同や順序崩れが減るかどうかである。

---

## 前提

今回 instruction に追加した再発防止は、
個別事象への対症療法ではなく、
次の 4 ゲートで構成される。

1. 状態層ゲート
2. completed condition ゲート
3. 依存順ゲート
4. 単発確認 / 継続確認 分離ゲート

したがって、
効果検証もこの 4 ゲートごとに行うのが自然である。

---

## 結論

検証は、
次の 1 ケースを観測対象にするのが最小で十分である。

### 検証ケース

`instruction 更新を含む task を 1 件扱い、
その task 提案 → completed 判定 → daily review 配置
までを通しで観測する`

今回の第一ケースとしては、

`ADAM_MMDD 再開識別子ルール`

を使うのが自然である。

理由:
- repo 反映はすでに起きている
- runtime 反映は未完了であり、誤判定しやすい
- Day0 / Day1 の依存順ミスも実際に発生した
- 単発確認と継続確認の分離も見やすい

つまり、
今回の実ケース自体が
4ゲート検証の最小テストケースになる。

---

## 検証対象

### ケース名

`ADAM_MMDD 再開識別子ルールの runtime 反映確認`

### 対象判断

- repo 反映の扱い
- runtime 未確認の completed 判定
- Day0 / Day1 配置
- 単発確認 / 継続確認の分離

---

## 検証観点

### 1. 状態層ゲート

確認すること:
- repo / canonical / runtime を分けて説明できたか
- repo 更新済みを runtime 反映済みと誤認しなかったか

pass 条件:
- `repo反映済み / 実運用反映未完了`
  と明示できる

fail 条件:
- repo 更新だけで completed 扱いする

---

### 2. completed condition ゲート

確認すること:
- `何をしたか` ではなく
  `何が観測できたら閉じるか`
  を主語にできたか

pass 条件:
- `次の新スレッド再開時に ADAM_MMDD を再開識別子として読めること`
  を completed condition として扱える

fail 条件:
- instruction 追記済みを理由に閉じる

---

### 3. 依存順ゲート

確認すること:
- 明日やる task として、
  intake routing より前に置くべきものを前に置けたか
- Day0 / Day1 の配置が前提順に合っているか

pass 条件:
- `ADAM_MMDD runtime 反映確認`
  を intake routing より前に置ける
- 明日やるなら Day0 に置ける

fail 条件:
- runtime 反映確認が後段の intake routing より後ろに置かれる

---

### 4. 単発確認 / 継続確認 分離ゲート

確認すること:
- 1 回の runtime 確認と、
  継続運用で守れることを分けて扱えるか

pass 条件:
- 今回はまず単発確認として閉じ、
  必要なら継続観測 task を別で置くと説明できる

fail 条件:
- 1 回確認しただけで恒久的に成立したとみなす

---

## 実施手順

1.
対象ケースを 1 件選ぶ

2.
そのケースについて、
現在の task 提案を確認する

3.
completed condition を言語化する

4.
Day 配置が依存順に合っているか確認する

5.
4 ゲートごとに pass / fail を記録する

6.
必要なら instruction または operations の補正要否を判断する

---

## 記録フォーマット

- ケース名
- 期待された判断
- 実際の判断
- 状態層ゲート: pass / fail
- completed condition ゲート: pass / fail
- 依存順ゲート: pass / fail
- 単発確認 / 継続確認 分離ゲート: pass / fail
- 再発した混同
- 次の補正要否

---

## 期待する結果

理想は、
4 ゲートすべてが pass することではなく、
少なくとも

- ミスが発生したときに
- どのゲートを通せば止まれたか

が明確になることである。

つまり、
この検証は
`正解できたか`
だけを見るものではなく、
`判断機構が改善したか`
を見るものである。

---

## 次に自然なタスク

1.
`ADAM_MMDD 再開識別子ルールの runtime 反映確認` を
この検証フォーマットで観測する

2.
必要なら
検証結果を analysis に返す

3.
4 ゲートのうち弱い箇所だけを追加補正する

---

## 判断

今回の instruction 追加が効果的かどうかは、
個別ルールが増えたかではなく、
次の実判断で

- 層混同
- completed 条件の取り違え
- 依存順崩れ
- 単発確認と継続確認の混同

が減るかで判断すべきである。

そのため、
`ADAM_MMDD 再開識別子ルール`
を第一検証ケースとして、
4 ゲートごとに pass / fail を記録する方式を採用するのが自然である。
