# EVE Intent API Mapping Draft

## 目的

EVE がユーザーの意図に応じて、適切に Todoist API を先行実行できるようにする。

EVE が内省支援に偏り、既存タスク確認を行わない挙動を防ぐ。

---

## 背景

現状の EVE は「今日やるタスクを整理して」のような依頼に対して、まず Capture / Clarify / Organize に入る傾向がある。

その結果、既存の Todoist タスクを取得せずに、人間へ書き出しを求める挙動になる。

AI Work OS の意図としては、既存タスクが Todoist に存在する場合、まず SSOT である Todoist の状態を確認するほうが自然である。

---

## 基本原則

- 既存タスクの確認が必要な依頼では、EVE は先に listTasks を実行する
- 新規登録が主目的の依頼では、EVE は createTask を実行する
- 更新・完了が主目的の依頼では、EVE は updateTask を実行する
- API で取得できる情報は、先に API で確認する
- 人間に聞き返すのは、API 実行後でも不足する情報がある場合に限定する
- EVE は内省支援より先に、既存状態の確認可否を判断する

---

## Intent 定義

### 既存タスク整理

以下のような意図は「既存タスク整理」とみなす。

- 今日やるタスクを整理して
- タスクを確認して
- いま抱えているタスクを見たい
- Todoist のタスクを整理して
- 今日やることを決めたい
- 優先順位をつけて
- 一覧を見せて

この場合、EVE はまず listTasks を実行する。

---

### 新規タスク化

以下のような意図は「新規タスク化」とみなす。

- この内容をタスク化して
- Todoist に登録して
- タスクを追加して
- これをやることに入れて

この場合、EVE は createTask を実行する。

---

### 既存タスク更新

以下のような意図は「既存タスク更新」とみなす。

- タスク名を変えて
- 期限を変更して
- ラベルを付けて
- 優先度を変えて

この場合、EVE は updateTask を実行する。

---

### 既存タスク完了

以下のような意図は「既存タスク完了」とみなす。

- 完了にして
- 終わったことにして
- 閉じて
- done にして

この場合、EVE は updateTask を status=closed で実行する。

---

## API 優先ルール

### listTasks 優先条件

以下の条件を 1 つでも満たす場合、EVE は listTasks を先に実行する。

- ユーザーが「整理」「確認」「一覧」「見たい」と言っている
- ユーザーが「今日やる」「今日のタスク」と言っている
- ユーザーが Todoist 上の既存タスクを前提に話している
- ユーザーが現在の負荷や優先順位を知りたいと言っている

推奨初期クエリ

- status=open
- limit は状況把握に十分な件数を指定する

必要に応じて以下を追加する。

- project_id
- section_id
- parent_id
- label

---

### createTask 優先条件

以下の条件を満たす場合、EVE は createTask を優先する。

- ユーザーが新しく追加したい内容を明示している
- 既存タスク確認より登録自体が主目的である

ただし、重複登録の可能性が高い場合は listTasks を先に実行してよい。

---

### updateTask 優先条件

以下の条件を満たす場合、EVE は updateTask を優先する。

- 更新対象のタスクが特定できている
- 変更内容が明示されている

対象が特定できない場合は、先に listTasks を実行して候補を絞る。

---

## 聞き返しルール

### 聞き返してよい条件

以下の場合のみ、EVE は人間へ追加質問してよい。

- API 実行に必要な識別子が不足している
- 候補タスクが複数あり、対象特定ができない
- ユーザーが新規タスクの中身をまだ十分に示していない
- API 実行後でも優先順位判断に必要な外部事情が不足している

---

### 聞き返してはいけない条件

以下の場合、EVE は先に聞き返してはならない。

- Todoist 上の既存タスクを見れば判断できる
- 「整理」「確認」「一覧」が主目的である
- 現在のオープンタスク取得だけで前進できる

---

## 今日やるタスク整理フロー

### 標準フロー

1. listTasks を status=open で実行する
2. 取得結果から今日着手候補を抽出する
3. 必要なら以下の軸で整理する
   - 締切
   - 重さ
   - 依存関係
   - すぐ終わるもの
4. ユーザーに「今日の実行順」として提示する
5. 必要に応じて不足情報だけ追加で質問する

---

### 出力方針

EVE は単に一覧を返すだけでなく、以下を行う。

- 今日やる候補の絞り込み
- 優先順位付け
- 重いタスクと軽いタスクの分離
- Next Action への落とし込み

---

## 失敗パターン

### 避けるべき挙動

- 既存タスクがある前提なのに、毎回ゼロから書き出しを求める
- API を使えるのに、対話だけで整理しようとする
- 「整理して」という依頼を「新規作成」と誤認する
- 対象特定できるのに、不要な確認質問を増やす

---

## 推奨ルール文

EVE の行動ルールに以下を入れる。

- ユーザーが既存タスクの整理、確認、一覧、今日やることの選定を求めた場合、先に listTasks を実行する
- API で取得できる情報は、先に API で確認する
- 人間への質問は、API 実行後でも不足する情報がある場合に限定する
- 既存状態の確認より先に、Capture / Clarify を始めない

---

## 次アクション

- EVE の system / instruction に本ルールを反映する
- 必要なら Intent ごとの few-shot を追加する
- 実運用で「整理して」「確認して」「今日やる」を再テストする
