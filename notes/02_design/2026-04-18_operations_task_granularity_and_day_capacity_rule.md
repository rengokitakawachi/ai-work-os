# 2026-04-18 operations_task_granularity_and_day_capacity_rule

## 目的

operations task の粒度と、
Day をどう扱うかの運用ルールを固定する。

本メモは、

- task が大きすぎて閉了条件が曖昧になる
- Day を 1 task 固定 slot のように誤解しやすい
- 1日の実行可能量をどう見るかが未固定

という問題を減らすため、
`docs/17_operations_system.md`
へ上げる前の最小 design を定義することを目的とする。

---

## 結論

operations では、
次を基本ルールとして扱う。

### 1. task 粒度

1 task は、
**0.5〜1.5時間程度で一区切りつく明確な作業単位**
を基本とする。

### 2. Day の意味

Day は、
**1 task 固定の slot ではなく、複数 task を置ける箱**
とする。

### 3. 1日実行枠

1日の想定実行量は、
**まずは 2 時間前後を目安**
とする。

ただし、
これは固定仕様ではなく、
その日の可処分時間や状況に応じて流動的でよい。

### 4. 知見蓄積

1日実行量の最適値や task 粒度の実用値は、
将来的に review / dev_memo / 実績から
**運用ノウハウとして蓄積する対象**
とする。

---

## なぜ必要か

現行の operations では、

- `〜を整理する`
- `〜を進める`
- `〜を検討する`

のような大きい task が入りやすい。

この状態だと、

- 完了条件が曖昧になる
- daily review で完了判定しにくい
- 繰越が増えやすい
- Todoist projection でも見え方が粗くなる

したがって、
operations task は
theme や plan ではなく、
**今日の着手で一区切りつく作業単位**
に寄せる必要がある。

---

## task 粒度ルール

### 基本

- 1 task = 1つの明確な作業単位
- 1回の着手で一区切りつくことを基本とする
- 目安は 0.5〜1.5時間とする

### 分割条件

以下に当てはまる場合は分割を検討する。

- 2時間を超えそう
- 完了条件が一文で言えない
- 複数の成果物や判断が混ざっている
- `AしてBしてCする` になっている
- 読んだだけで着手手順が分からない

### 避ける表現

- `整理する`
- `進める`
- `検討する`
- `対応する`

これらを完全に禁止するわけではない。

ただし、
それだけでは task が大きくなりやすいため、

- 何を
- どこまで
- 何をもって一区切りか

が task 名または notes から分かる形にする。

---

## Day 容量モデル

### Day は箱である

Day0〜Day6 は、
各日に複数 task を載せられる箱とする。

したがって、

- Day = 1件

ではなく、

- Day = その日の実行枠

と解釈する。

### 置き方の目安

最小運用では、
1日の想定実行量を 2時間前後とみなし、
その範囲で複数 task を載せてよい。

例:

- 0.5h + 1.0h
- 1.0h + 1.0h
- 1.5h 単体
- 0.5h + 0.5h + 0.5h

ただし、
実際の可処分時間は流動的なので、
毎日必ず 2時間埋める必要はない。

---

## theme / plan / operations task の区別

### theme

数日〜数週で前進させる論点束。

### plan

2〜4週間程度の重点テーマ。

### operations task

今日の着手で一区切りつく作業単位。

したがって、

- theme をそのまま active に入れない
- plan をそのまま task 名にしない
- operations task は theme を分解した単位にする

---

## completion 判定との関係

粒度ルールを入れる目的は、
単に task 数を増やすことではない。

重要なのは、

- 完了
- 継続
- 分割し直し
- 次日へ繰越

を判断しやすくすることにある。

このため、
完了条件が言えない task は、
active に入れる前に分割候補として扱う方が自然である。

---

## Todoist projection との関係

Todoist は execution view であるため、
operations task の粒度が粗すぎると見え方が悪くなる。

逆に、
0.5〜1.5時間程度の単位で置けると、

- 今日やることが見やすい
- close しやすい
- 繰越判断もしやすい

したがって、
この粒度ルールは Todoist projection の運用安定にも効く。

---

## 固定すること / まだ固定しないこと

### 今固定すること

- 1 task は 0.5〜1.5時間程度を目安にする
- Day は複数 task を置ける箱とする
- 1日2時間前後を仮の目安とする

### まだ固定しないこと

- 毎日必ず何時間使うか
- 1日何件まで入れるかの固定上限
- 0.5h / 1h / 1.5h の正式分類
- 実績時間の厳密記録場所

これらは、
運用実績から知見として蓄積していく。

---

## docs 反映候補

`docs/17_operations_system.md`
へ上げるなら、
次のような要点で十分である。

- operations task は 1つの明確な作業単位とする
- 目安は 0.5〜1.5時間程度とする
- Day0〜Day6 は複数 task を置ける箱とする
- 1日の想定実行量は仮に 2時間前後を目安とする
- 実際の実行量は固定せず、運用知見として蓄積する

---

## 判断

operations の粒度ルールは、
運用の癖ではなく
**短期実行順正本の品質条件**
として扱うのが自然である。

したがって、
最小ルールとして

- task は 0.5〜1.5時間程度
- Day は複数 task を置ける箱
- 1日2時間前後は仮目安
- 実用値は知見蓄積対象

を固定し、
次段で docs へ昇格させるのがよい。
