# 2026-04_phase1_schedule_proposal_and_outlook_write

## phase

Phase 1: Prototype / MVP

## 位置づけ

Phase 1 の foundation plan に続く第2段階の plan。

Todoist 一覧取得、
Outlook 予定表読取、
空き時間把握の基盤を前提に、
優先順位づけしたタスクをもとにスケジュール案を提示し、
承認後に Outlook へ書き込む流れを成立させることを目的とする。

本 plan は、
EVE の最小実行系を
「読める状態」から
「提案して反映できる状態」へ進めるための中位計画とする。

---

## 目的

EVE が
Outlook の空き時間と Todoist タスクをもとに
実行可能なスケジュール案を提示し、
承認された内容を Outlook に反映できる状態を作る。

---

## 背景

Phase 1 の foundation では、
EVE がタスクと予定を同時に把握できる状態を目指す。

ただし、
把握できるだけでは日常運用の価値はまだ弱い。

実際に必要なのは、
空き時間に対して優先順位づけされたタスクを当てはめ、
実行可能なスケジュール案を提示し、
承認後に予定へ反映できることである。

そのため、
本 plan では
read 中心の foundation の次段階として、
proposal と write を成立対象にする。

---

## スコープ

この plan で扱うもの

- タスク優先順位づけの最小ルール整理
- 空き時間とタスクの突合ロジック整理
- スケジュール案の提示形式整理
- 承認フローの最小設計
- 承認後の Outlook 予定書き込み
- proposal → approval → write の最小ループ成立

この plan で扱わないもの

- GitHub 正本との同期
- Todoist 操作結果の正本反映
- MindMeister 連携
- Teams 通知自動化の完成
- Obsidian の本格連携
- 高度な最適化アルゴリズム
- 複雑な競合解決や再スケジューリング

---

## 完了条件

- タスク優先順位の最小ルールを定義できる
- Outlook の空き時間へタスクを当てはめた案を生成できる
- ユーザーへスケジュール案を提示できる
- 承認 / 非承認の分岐を扱える
- 承認された案を Outlook に書き込める
- Phase 1 の MVP として「提案して反映する」最小ループが成立する

---

## 主要論点

### 1. 優先順位づけの基準

- Todoist の priority をどう使うか
- due date をどの程度重く扱うか
- 緊急性と重要性をどう簡易化するか
- 手動 override を前提にするか

### 2. スケジュール案の作り方

- 空き枠に 1件ずつ割り当てるか
- 当日優先か翌日以降も含めるか
- 長いタスクを分割するか
- 休憩や移動バッファをどう扱うか

### 3. 承認フロー

- 提示単位を 1件ごとにするか、全体案にするか
- 部分承認を認めるか
- 修正再提案をどう扱うか

### 4. Outlook 書き込みの範囲

- 新規イベントとして書くか
- 仮予定として書くか
- タイトルや説明に何を入れるか
- 書き込み失敗時の扱いをどうするか

---

## 現時点の仮方針

- 優先順位づけは単純ルールから始める
- 最初は当日または直近の空き枠を対象にする
- スケジュール案は全体案として提示する
- 承認後のみ Outlook に書き込む
- write は最小限のイベント作成から始める
- 複雑な最適化は後回しにする
