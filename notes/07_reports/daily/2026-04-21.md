## 重要な意思決定

- issue routing の completed 判定は、実装完了ではなく route 多様性観測と運用妥当性確認で見る
- `route_to: operations` は即 active 化ではなく rolling 比較対象化を意味する
- `keep / future / archive` は、それぞれ `現 phase 内の軽い保留 / 今は対象外 / 役目終了` として区別する
- 反映系 task は `repo反映 / 正本反映 / 実運用反映` を分離して完了判定する
- マイGPT運用では、`ADAM_MMDD` はスレッド名固定ではなく再開識別子として扱う
- ただし `ADAM_MMDD` ルールは repo 反映済みでも runtime 反映確認は未完了であり、翌日の Day0 に残す

## 学び / 気づき

- 「反映した」を 1 語で扱うと、repo 更新済みと runtime 反映済みを再び混同しやすい
- issue routing の完成はコード完成ではなく、送付先多様性と送付先ごとの意味が説明できることにある
- `future` と `keep` の境界は、status や phase の違いだけでなく「今やる対象か」を言葉で説明できるかが重要
- operations candidate を rolling 前段と理解すると、routing と operations の責務境界が崩れにくい
