## 重要な意思決定

- active-first execution では、stale な active 先頭 task を見つけた場合、reroll や新規実行の前に active 正本の整合回復を行う
- reroll before execution は毎ターン無条件に行うものではなく、新規候補発生時の再配置ルールとして扱う
- 会話中に新規課題または新規作業候補が出た場合でも、その場で横入り実行せず、先に
  `議論 / issue / operations candidate`
  を判定する
- operations candidate の場合は、保存や実行の前に active / next / future の配置提案を行う
- decision は EVE で重要な履歴対象になりうるため、まず ADAM で最小運用モデルを試す

## 学び / 気づき

- active-first execution は、少なくとも「stale な active を正してから先頭 task を進める」基本ケースでは成立する
- reroll を原則化しても、発火条件を絞らないと実運用で過剰になりやすい
- ADAM は複数論点や会話の勢いに引かれると operations 原則を忘れやすく、停止ルールを instruction に明文化しないと崩れやすい
- stale active の整合回復は reroll と別物であり、「正本修復」の前処理として切り分けるのが自然
- decision 論点は docs / issue / design / plan / operations / dev_memo を横断して参照できる構造が必要になりそうだが、現時点では next に留めるのが妥当
