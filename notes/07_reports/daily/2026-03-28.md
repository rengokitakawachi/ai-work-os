## 重要な意思決定

- GitHub ユーザー名変更は、単なる命名変更ではなく運用変更として扱う
- GitHub ユーザー名変更は後回しにすると依存箇所が増えるため、早期対応候補とする
- issue の優先順位は「土台になるもの」「後回しでコストが増えるもの」「混在状態を減らすもの」を優先する
- 旧開発メモフォルダの移行と README 更新は、混在状態を減らす効果が高いため前倒しする
- 標準開発フローは step ごとに 判断内容 / 保存先 / 次アクション を同時に決める形で整理する
- 旧開発メモ移行は棚卸し先行ではなく、先に新フォルダへ複写してから扱う方針を採用した
- 棚卸しと分類はインテークレビュー機能の後に行う方針とした
- 検証済み複写が完了したため、旧 `exploration/memo` は削除してよいと判断した

## 学び / 気づき

- 開発メモを issue 候補抽出の母集団として扱うと、見落としていた構造課題をかなり拾える
- issue が増えてきた段階では、個別整理よりも execution order の設計が重要になる
- 混在状態を減らす作業は、抽象的な土台づくりと同じくらい運用効率に効く
- 複数ファイルの移行では、作業完了と検証完了を分けて扱わないと取りこぼしが起きやすい
- SHA 照合を最後に入れることで、目視では見落とす差分も検出できる
