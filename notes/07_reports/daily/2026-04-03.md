## 重要な意思決定

- 新スレッド再開時は handover 探索から開始し、handover を読まずに状況把握を始めない
- 再開時の入口は handover、仕様判断の正本は docs、短期実行順の正本は operations とする
- `code bulk` は便利機能ではなく、repo-resource の access layer 一貫性と再開効率のための必要論点として扱う
- 明日の優先順位は Todoist ではなく operations を正として決める
- daily review は、日報保存だけで完了にせず、content 抽出確認と operations 確認までを完了条件とする
- `review` と `routing` は分けて扱い、`intake routing` と `issue routing` を共通骨格として整理する
- 今の ADAM 開発は単発の開発支援ではなく、AI Work OS 全体の骨格開発として扱う
- 上記の意味づけを `Phase 0: Common Operating Model Foundation` として plan 化する
- `flow check` を、review とは別に、運用の滞留 / 停滞 / 目詰まりを確認する機能候補として保持する

## 学び / 気づき

- handover ルールは「読む」だけでなく、「その後どの正本へ接続するか」まで定義しないと再開品質が安定しない
- `docs / notes / code` の操作モデルを揃える思想に対し、code だけ bulk がないことは実運用上の違和感として現れている
- operations に 1 行追加するだけでも、翌日の着手順はかなり明確になる
- Todoist にテンプレート由来のノイズが多い場合、短期開発の実行順は operations を見た方が精度が高い
- instruction を読んでいても、実行フローが固定化されていないと取りこぼしは起きる
- 再発防止には「覚える」より「完了条件の固定」が効く
- `review` より `routing` として捉えた方が責務分離が美しくなる
- ADAM の開発フローを共通 OS モデルとして見直すと、EVE への展開可能性がかなり見えやすくなる
- issue の収集だけでなく、issue を plan / operations / dev_memo へ流す後段の運用が必要だと整理できた
- routing / planning / review だけでなく、「どこで流れが止まっているか」を見る `flow check` 層も必要だと見えてきた
