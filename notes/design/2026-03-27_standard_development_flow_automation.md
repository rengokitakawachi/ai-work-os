# 2026-03-27 Standard Development Flow Automation

## 目的

課題発見から実装までの標準開発フローを定義する。

同時に、フローに基づく判断を ADAM が自動で行える状態を設計する。

---

## 背景

現状では、開発フローが一貫していない。

そのため、以下の問題が発生している。

- ファイルが散らかる
- どこを見ればよいか分かりにくい
- 同じ確認を繰り返しやすい
- 作業効率が落ちる

特に、課題、検討、設計、実装の保存先や次アクションの判断が場面ごとにぶれやすい。

---

## 問題

現在の課題は、標準フローがないことだけではない。

標準フローがないため、以下の判断が毎回手動依存になっている。

- 課題の種類判定
- 保存先判定
- 次アクション判定
- 実装に進んでよいかの判定

この状態では、運用が人の都度判断に依存し、再現性が低い。

---

## 目標

以下を満たす状態を目標とする。

- 課題の種類ごとに保存先が決まる
- 課題の種類ごとに次アクションが決まる
- 課題発見から実装まで毎回同じ順序で進められる
- ADAM が現在フェーズを判定できる
- ADAM が保存先と次アクションを自動判定できる

---

## 基本方針

- フローは人向けの説明ではなく、ADAM の実行ルールとして定義する
- 判断基準を先に固定し、保存先と次アクションを後から決めるのではなく一体で設計する
- docs / notes / code / operations の責務分離を維持する
- 標準フローは課題の種類ごとの分岐を含む

---

## 標準フロー案

### 1. 課題発見

入力された論点を受け取り、まず課題として認識する。

### 2. 課題整理

以下を構造化する。

- title
- category
- description
- root_cause
- impact
- scope
- stakeholders
- candidate_solution

保存先の第一候補は ideas とする。

### 3. 種類判定

課題を以下のような種別に分類する。

- bug
- ux
- api
- sync
- architecture
- ops
- idea

### 4. 保存先判定

課題の種類と成熟度に応じて保存先を決める。

想定ルール

- 未整理の課題: ideas
- 仕様草案が必要: design
- 短期の実行管理が必要: operations
- 仕様として確定済み: docs 反映候補
- 実装対象: code

### 5. 次アクション判定

課題の種類と成熟度に応じて次アクションを決める。

想定ルール

- 課題が曖昧: ヒアリング継続
- 課題が明確: ideas へ保存
- 設計が必要: design 草案作成
- 仕様確認が必要: docs 読取
- 実装条件が揃った: code 確認 / 実装
- 後でやる必要がある: operations 記録

### 6. 設計整理

仕様草案が必要な場合は notes/design に整理する。

### 7. 仕様確認

docs を取得し、既存仕様との整合を確認する。

### 8. 実装判断

以下を満たす場合のみ実装へ進む。

- 課題が構造化されている
- 関連 docs を取得済み
- 仕様との整合が確認できている
- design が必要な場合は作成済み

### 9. 実装

code は docs に従属して更新する。

---

## 自動化対象

ADAM が自動化すべき判断は以下とする。

- 課題の種類判定
- 保存先判定
- 次アクション判定
- 現在フェーズ判定
- 実装可否判定

---

## 正本の役割

役割分離は以下を前提とする。

- docs: SSOT
- notes/ideas: 課題記録
- notes/design: 仕様草案
- notes/operations: 短期実行管理
- code: 実装

---

## 期待効果

- ファイル配置が安定する
- 課題から実装までの見通しがよくなる
- 同じ確認の繰り返しが減る
- ADAM の判断が再現可能になる
- 開発運用の効率が上がる

---

## 残論点

- 課題カテゴリごとの詳細分岐ルール
- 保存先判定の具体条件
- operations に送る条件
- docs 更新判断の条件
- 実装開始条件の厳密化
- ADAM / EVE の役割分担との接続

---

## 次アクション

次に整理すべき内容は以下とする。

1. 課題カテゴリごとのルーティング表を作る
2. 保存先判定ルールを表形式で固定する
3. 実装開始条件を明文化する
4. ADAM の instruction に判断ルールを反映する
