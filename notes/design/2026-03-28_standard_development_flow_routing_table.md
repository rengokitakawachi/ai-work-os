# 2026-03-28 Standard Development Flow Routing Table

## 目的

標準開発フローの各ステップに対して、判断内容・保存先・次アクションを対応付ける。

ADAM が課題発見から実装判断までを一貫した順序で扱えるようにする。

---

## 前提

- docs は SSOT とする
- notes は補助レイヤーとする
- code は docs に従属する
- operations は短期実行管理とする
- design は docs 直前の仕様草案とする

---

## 基本方針

- まず現在の入力がどの段階にあるかを判定する
- 判定と同時に保存先と次アクションを決める
- 1つ先の行動が明確にならない状態で実装へ進まない
- docs 未取得状態では仕様判断と実装判断を行わない
- design が必要な論点は code に進める前に design へ送る

---

## 対応表

| step | 判断内容 | 保存先 | 次アクション |
|---|---|---|---|
| 1. 課題発見 | 入力が新規論点か、既存論点の追記かを判定する | 未保存 または inbox | 未整理なら課題整理へ進む |
| 2. 課題整理 | title / category / description / impact / scope / stakeholders / candidate_solution を構造化できるか判定する | ideas | 構造化不足ならヒアリング継続、構造化できたら種類判定へ進む |
| 3. 種類判定 | bug / ux / api / sync / architecture / ops / idea のどれかを判定する | ideas | 種類に応じて保存先判定へ進む |
| 4. 保存先判定 | 論点の成熟度と用途から、ideas / design / operations / docs 参照 / code 確認のどこに送るべきか判定する | ideas, design, operations のいずれか | 次に必要な作業を1つに絞る |
| 5. 次アクション判定 | 今必要なのがヒアリング、設計、仕様確認、実装確認、実装、後回し記録のどれかを判定する | operations または design または継続対話 | 実際の次工程へ進む |
| 6. 設計整理 | 仕様草案が必要か判定する | design | 必要なら design 草案作成、不要なら仕様確認または operations へ進む |
| 7. 仕様確認 | 関連 docs を取得済みか、既存仕様と矛盾がないかを判定する | docs は読取、必要なら design | docs 未取得なら取得、整合確認後に実装判断へ進む |
| 8. 実装判断 | 実装条件が揃っているか判定する | operations または design | 条件不足なら不足解消、条件充足なら code 確認へ進む |
| 9. code 確認 | 既存実装の位置、責務、変更範囲、影響範囲を判定する | code 読取、必要なら design / operations | 差分最小の変更案を作る |
| 10. 実装 | docs と整合する範囲で変更する | code | 実装結果確認、必要なら design / docs 差分整理へ進む |
| 11. 実装後整理 | docs 差分、notes 反映、operations 更新が必要か判定する | operations, design, reports | 未反映差分を整理し、次アクションを更新する |

---

## 保存先判定ルール

### ideas

以下の場合に保存する。

- 課題は認識できたが、まだ未整理である
- 論点を issue として残す価値がある
- 設計や実装に直行する前に構造化が必要である

### design

以下の場合に保存する。

- 仕様草案が必要である
- docs に影響する可能性がある
- 保存先や運用ルールの設計が必要である
- 実装前に構造を固定する必要がある

### operations

以下の場合に保存する。

- 近々実行するタスクとして追跡が必要である
- 優先順位管理が必要である
- 今日 / 今週の実行順に載せる必要がある

### docs 読取

以下の場合に必須とする。

- 仕様確認が必要である
- docs と矛盾しないことを確認する必要がある
- code を更新する可能性がある

### code 読取

以下の場合に進む。

- 実装判断が通った
- 既存コードの責務と配置を確認する必要がある
- 変更対象を特定する必要がある

---

## 実装開始条件

以下をすべて満たす場合のみ、実装へ進む。

- 課題が構造化されている
- 次アクションが明確である
- 関連 docs を取得済みである
- docs との整合を確認済みである
- design が必要な場合は草案がある
- 変更対象 code の位置と責務が分かっている

---

## ルーティングの簡易規則

### 課題が曖昧な場合

- 保存先は対話継続または ideas
- 次アクションはヒアリング継続

### 課題が明確だが設計が未整理の場合

- 保存先は ideas または design
- 次アクションは設計整理

### 設計は明確だが仕様確認が未了の場合

- 保存先は design
- 次アクションは docs 読取

### 仕様確認済みで実装条件が揃った場合

- 保存先は operations
- 次アクションは code 確認 → 実装

### 今すぐ実装しないが忘れたくない場合

- 保存先は operations または ideas
- 次アクションは優先順位付け

---

## ADAM の判断順

ADAM は毎回以下の順で判断する。

1. これは新規論点か、既存論点の継続か
2. これは構造化済みか
3. 種類は何か
4. 今置くべき保存先はどこか
5. 次アクションは何か
6. docs 確認が必要か
7. design が必要か
8. 実装に進めるか

---

## 現時点の結論

- 課題発見から実装までを step 単位で管理する
- 各 step で 判断内容 / 保存先 / 次アクション を同時に決める
- design と docs 確認を飛ばして code に進まない
- operations は短期の実行管理として使う
- ideas は課題の入口、design は仕様草案、docs は正本、code は実装として分離する
