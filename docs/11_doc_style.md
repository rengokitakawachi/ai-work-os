# 11_Doc_Style: ドキュメント書式ルール

## 1. 目的

本ドキュメントは、AI Work OS における docs の記述ルールを定義する。

目的は以下の3点

- AIが安定して構造を解析できること
- 差分（Diff）が最小になること
- 人間とAIが共同で編集しても崩れないこと

---

## 2. 適用範囲

対象は docs 配下のすべてのドキュメントとする。

notes は補助レイヤーであり、本ルールの直接対象外とする。
ただし、docs との役割分離の基準として参照する。

---

## 3. 基本原則

SSOT = docs

構造は固定しすぎず、意味の一貫性を優先する

差分最小で更新する

未確定事項は docs に書かない

実装は docs に従う

---

## 4. レイヤー定義

### 4.1 docs

- 確定情報を記述する
- SSOT として扱う
- AIは最優先で読む

### 4.2 notes

- 未確定情報を記述する
- 思考過程を保持する
- docs の正本にはしない

### 4.3 code

- 実装を保持する
- docs に従属する
- 仕様の正本にはしない

### 4.4 loop

- 更新手順を定義する
- AIと人間の協働手順を扱う

---

## 5. 見出しルール

### 5.1 見出し階層

使用する見出しは以下のみ

- # 大項目
- ## 中項目
- ### 小項目

#### は使用しない

---

### 5.2 見出し命名

- 名詞で統一する
- 動詞を避ける
- 簡潔にする
- 同義語を増やさない

例

正
- 目的
- 適用範囲
- 基本原則
- 見出しルール
- 更新ルール

誤
- この文書について
- 補足説明
- 何をするか
- なぜ重要か

---

## 6. セクション構造

各ドキュメントは以下の順序を基本とする。

- 目的  
- 前提 / 定義（必要な場合のみ）  
- 基本原則  
- 処理（フロー）  
- ルール（制約・禁止事項）  
- 補足  

不要なセクションは省略可とする。  
ただし、目的は原則として省略しない。

---

## 7. 箇条書きルール

### 7.1 記号

- ハイフン「-」を使用する
- 全角記号を使わない

---

### 7.2 文体

- 1行1意味とする
- 短文で記述する
- 主観を避ける
- 断定で書く

例

正
- Taskは30〜120分で完了する単位
- Next Actionは動詞で始める

誤
- Taskは基本的に短時間で終わるように設計されるべきである
- Next Actionについてはケースによって変わる

---

## 8. コード表記ルール

### 8.1 基本

コードはコードブロックで記述する。

---

### 8.2 ネスト禁止

コードブロック内にコードブロックを入れない。

---

### 8.3 JSON表記

JSONは整形して記述する。

---

### 8.4 例外

出力制約によりコードブロックが使用できない場合

- インデントで代替する
- 構造維持を優先する

---

## 9. 命名規則

### 9.1 ファイル名

形式

NN_name.md

例

- 01_concept.md
- 03_api_spec.md
- 11_doc_style.md

---

### 9.2 API命名

形式

/api/{resource}/{action}

例

- /api/task/create
- /api/docs-read
- /api/notes-write

---

## 10. 空行ルール

- セクション間は1行空ける
- 箇条書きは詰める
- 過剰な空行を入れない

---

## 11. 更新ルール

AIおよび人間は以下を厳守する。

既存の設計意図を消さない

必要な情報は残す

不要な情報のみ削除する

差分最小で追記・修正する

構造変更は必要な場合に限定する

構造変更時は一貫性の向上を目的とする

---

## 12. 禁止事項

- 推測による追記
- 未確定事項の docs 記載
- 同じ意味の見出しの乱立
- 文体の混在
- 責務外の内容の混入
- notes に書くべき内容の docs 記載

---

## 13. AI向けルール

AIは以下を必ず守ること。

1. 更新前に対象ドキュメント全文を読む
2. 関連ドキュメントも必要に応じて読む
3. 既存の設計意図を優先して残す
4. 部分修正で済む場合は全体を書き換えない
5. 未確定事項は docs に書かない
6. 必要に応じて notes に記録する

---

### 13.1 Docs取得方法（必須）

AIは docs を直接推測してはならない。  
必ず API を使用して取得する。

ベースURL

https://ai-work-os.vercel.app

---

使用するAPI

GET /api/docs  
・docs一覧を取得する  

GET /api/docs-read?file=FILENAME  
・指定ファイルの本文を取得する  

---

例

https://ai-work-os.vercel.app/api/docs  
https://ai-work-os.vercel.app/api/docs-read?file=11_doc_style.md

---

### 13.2 行動ルール

- docsは必ずAPI経由で読む
- 過去の会話で代替しない
- 不明な場合は再取得する
- 取得せずに編集しない

---

### 13.3 禁止事項

- 「アクセスできない」と判断する
- docs未取得での修正
- 推測による補完

---

### 13.4 接続前提

AIはこのAPIにアクセス可能である前提で動作する。

アクセスできないと判断してはならない。

---

## 14. 完成状態の定義

ドキュメントが正しく書かれている状態とは以下である。

- 見出し構造が統一されている
- 命名が一貫している
- 役割分離が守られている
- 差分が最小で更新されている
- AIが構造を誤解しない
- 人間が追記しても崩れない
