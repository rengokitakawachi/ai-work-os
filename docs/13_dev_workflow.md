# 13_dev_workflow.md

## 概要

本ドキュメントは、AI Work OS における開発フローおよび人と AI の役割分担を定義する。

本ワークフローに従うことで、仕様と実装の整合性を維持する。

---

## 基本方針

開発は Docs Driven Development に基づいて行う。

仕様（docs）を起点とし、実装と更新を循環させる。

---

## 全体フロー

開発は以下の流れで行う。

課題の発見  
↓  
notes に記録  
↓  
設計整理  
↓  
docs に反映  
↓  
実装  
↓  
docs 更新  

---

## フェーズ定義

### 1 課題の発見

機能追加、改善、バグなどの課題を認識する。

---

### 2 notes への記録

課題やアイデアを notes に記録する。

未整理の内容は notes/inbox に格納する。

---

### 3 設計整理

notes/design にて仕様案を整理する。

必要に応じて notes/decisions に意思決定を記録する。

---

### 4 docs への反映

仕様が確定した内容のみ docs に反映する。

docs は SSOT である。

---

### 5 実装

docs に基づいて実装を行う。

実装は docs と整合している必要がある。

---

### 6 docs 更新

実装後に docs と差異があれば修正する。

更新は差分最小で行う。

---

## AI の役割

AI は以下を担当する。

docs の読取

仕様理解

差分検出

修正案の生成

docs 更新の補助

---

## 人間の役割

人間は以下を担当する。

意思決定

最終判断

実装の実行

外部環境の操作

---

## AI 行動ルール

AI は次のルールを守る。

docs を必ず API 経由で取得する

docs を最優先の仕様とする

推測で仕様を変更しない

差分最小で修正する

更新前に対象 docs を再取得する

---

## notes と docs の関係

notes は検討メモであり、SSOT ではない。

docs は正式仕様である。

notes の内容を docs に反映する場合は、仕様として確定している必要がある。

---

## 更新原則

更新は小さく行う。

一度に大きな変更を行わない。

常に整合性を維持する。

---

## 最重要原則

SSOT = GitHub docs

Docs Driven Development

仕様  
↓  
実装
