# 13_dev_workflow.md

## 目的

本ドキュメントは、AI Work OS における開発フローおよび人と AI の役割分担を定義する。

本ワークフローに従うことで、仕様と実装の整合性を維持する。

---

## 前提 / 定義

開発は Docs Driven Development に基づいて行う。

仕様（docs）を起点とし、実装と更新を循環させる。

docs は SSOT である。

notes は検討メモであり、SSOT ではない。

---

## 基本原則

- docs を最優先の仕様とする
- 実装は docs に従う
- notes → docs → code の順で確定する
- 更新は差分最小で行う
- 未確定事項は docs に書かない

---

## 処理（フロー）

### 全体フロー

課題の発見  
↓  
notes に記録  
↓  
設計整理  
↓  
docs に反映  
↓  
実装  
↓  
docs 更新  

---

### フェーズ定義

#### 1 課題の発見

機能追加、改善、バグなどの課題を認識する。

---

#### 2 notes への記録

課題やアイデアを notes に記録する。

未整理の内容は notes/inbox に格納する。

---

#### 3 設計整理

notes/design にて仕様案を整理する。

必要に応じて notes/decisions に意思決定を記録する。

---

#### 4 docs への反映

仕様が確定した内容のみ docs に反映する。

docs は SSOT である。

---

#### 5 実装

docs に基づいて実装を行う。

実装は docs と整合している必要がある。

---

#### 6 docs 更新

実装後に docs と差異があれば修正する。

更新は差分最小で行う。

---

### docs 修正フロー（必須）

docs 修正時は必ず以下の手順で行う。

1  
GET /api/docs により docs 一覧を取得  

2  
対象 docs を /api/docs-read で取得  

3  
関連 docs を取得  

4  
差分確認  

5  
修正作成  

6  
必要に応じて再取得（最新化）  

7  
全文出力  

---

## AI の役割

AI は以下を担当する。

- docs の読取
- 仕様理解
- 差分検出
- 修正案の生成
- docs 更新の補助

---

## 人間の役割

人間は以下を担当する。

- 意思決定
- 最終判断
- 実装の実行
- 外部環境の操作

---

## ルール（制約・禁止事項）

### Docs取得ルール

- docs は必ず API 経由で取得する
- docs は /api/docs-read により本文取得する
- 過去の会話で代替しない
- 不明な場合は再取得する

---

### 停止条件

- docs 未取得の場合は修正出力を行ってはならない
- 取得済み docs を明示していない場合は処理を停止する

---

### 11準拠

- 11_doc_style.md を事前に取得する
- docs の記述は 11 に完全準拠する

---

### 出力ルール

- 必ず全文出力する
- 部分出力は禁止する
- コードブロック内で出力する
- コードブロックのネストは禁止する

---

### コード制約

- 現状コードは API 取得不可
- 構造把握のうえ全文出力する

---

## notes と docs の関係

notes は検討メモであり、SSOT ではない。

docs は正式仕様である。

notes の内容を docs に反映する場合は、仕様として確定している必要がある。

---

## 更新原則

更新は小さく行う。

一度に大きな変更を行わない。

常に整合性を維持する。

---
